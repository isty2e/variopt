# Preserve refinement accounting in Study terminal surfaces

Logical ID: var-ygko. Terminal Study/report/result surfaces must expose enough refinement accounting to explain what was proposed, what was evaluated, and what changed without conflating evaluation budget with refinement work.

## Acceptance Criteria

Terminal surfaces expose refinement accounting coherently; evaluation counts still mean evaluator calls; docs and tests distinguish proposed/refined/evaluated candidates; serialization/checkpoint paths either preserve the data or explicitly reject unsupported state.


## Notes

**2026-06-30T23:38:09Z**

Procedure hardening and self-review before implementation:
1. Ontology split: RunReport is the canonical terminal history; RunResult and NondominatedRunSurface are terminal projections over that history. Projection surfaces may preserve aligned provenance, but must not own refinement policy or evaluation semantics.
2. Boundary basis: evaluation_count remains evaluator-call/accounting cost. CandidateRefinement explains proposed vs evaluated candidate changes separately and must never increment or reinterpret evaluation_count.
3. Lean pressure: do not add a new public accounting wrapper, trace payload, or serializer. Add the smallest projection fields needed to avoid dropping RunReport.refinements.
4. Responsibility audit: terminal.py owns validation that refinement metadata aligns with the records/observations it stores; CandidateRefinement owns source/refined/path facts; Study.optimize owns projection threading from RunReport to RunResult.
5. Package taxonomy: all terminal-surface projection changes stay in artifacts/terminal.py; Study wiring stays in study/execution.py; CSA checkpoint serialization remains in CSA state modules and should not absorb terminal result serialization.
6. Serialization/checkpoint decision: CSA safe-boundary checkpoints serialize optimizer state, not Study terminal reports. Since terminal surfaces expose no to_dict/from_dict contract, docs should state that terminal result serialization is intentionally caller-owned for now rather than silently unsupported.
7. Rejected alternatives: do not attach refinement accounting to TraceEvent; do not add scalar-only refinement counters that hide changed paths; do not create separate best/frontier refinement DTOs before aligned provenance proves insufficient.
8. Edge-hunt targets: RunResult projection loss, NondominatedRunSurface.from_report loss, mismatched projection refinement lengths, mismatched refined candidates, all-None sentinel canonicalization, no-refinement allocation, and docs/checkpoint wording drift.
